Time Module — Roadmap to S-CORE v1.0
=====================================

.. rubric:: Document info

- **S-CORE v1.0 Roadmap**: `reference_integration docs <https://github.com/eclipse-score/reference_integration/tree/main/docs/s_core_v_1>`_
- **Module board**: `TIM - Time FT #57 <https://github.com/orgs/eclipse-score/projects/57>`_
- **PI tickets**: `#69 <https://github.com/eclipse-score/inc_time/issues/69>`_ (v0.8) · #3011 (v0.9) · #3013 (v0.10) · #3014 (v1.0) in ``eclipse-score/score``
- **Updated**: 2026-06-23

.. note::

   This is an initial draft plan. Targets will be refined as the module progresses through
   each release gate. Raise a PR against this file to propose changes.

Context
-------

The S-CORE v1.0 roadmap is structured around four release gates.
Each gate has **mandatory minimum deliverables**; teams are free to advance ahead
but the gate requirements are non-negotiable for the final release.

.. list-table:: Milestone Timeline
   :header-rows: 1
   :widths: 12 28 60

   * - Milestone
     - Period
     - Project Focus
   * - **v0.8**
     - 13 May – 13 Jul 2026
     - Requirements Engineering (PA2) + Architecture Design (PA3)
   * - **v0.9**
     - 14 Jul – 7 Sep 2026
     - Implementation — Detailed Design (PA4)
   * - **v0.10**
     - 8 Sep – 2 Nov 2026
     - Verification — Coverage & Inspection Completeness (PA5)
   * - **v1.0**
     - 3 Nov – 15 Dec 2026
     - Hardening & Release

**v0.8 is the baseline** for the Time module.
The actual v0.8 scope for TIM is intentionally narrow: the S-CORE project gate for v0.8 calls
for requirements and architecture complete, but the Time module is entering that gate with a
different priority — stabilise the public API, confirm the end-to-end functional chain, and
integrate into ``reference_integration`` CI
(`#68 <https://github.com/eclipse-score/inc_time/issues/68>`_ is the only sub-issue under the
v0.8 PI `#69 <https://github.com/eclipse-score/inc_time/issues/69>`_).
Existing general documentation (README, examples, informal PlantUML diagrams) and the three
pre-existing ``comp_req`` sets (time, time_daemon, time_slave) are carried over from prior work
but are **not formal v0.8 deliverables**.

This roadmap describes a realistic step-by-step plan that achieves all v1.0 gate requirements
by December 2026, treating v0.9 as the primary catch-up sprint for process artefacts.

**Feature scope for v1.0** — features marked ``deferred`` have no ``valid_from ≤ v1.0`` on their
``feat_req`` nodes and are automatically excluded from v1.0 coverage statistics; their component
artefacts (``comp_req``, ``comp_arc``, ``comp_dd_sta``) will be authored in a post-v1.0 PI.

Maturity Level Key
------------------

.. list-table::
   :header-rows: 1
   :widths: 8 18 74

   * - Level
     - Symbol
     - Meaning
   * - L0
     - ``—``
     - Not started
   * - L1
     - ``draft``
     - File/node exists; content incomplete or not yet consistent
   * - L2
     - ``valid``
     - sphinx-needs ``:status: valid``; all TBDs resolved; fully traceable
   * - L3
     - ``filed``
     - ``valid`` + inspection checklist submitted; all findings tracked with milestone ≤ v1.0
   * - L4
     - ``inspected``
     - ``:status: valid(inspected)``; all findings resolved; approved by a Committer
